### LLMWikiの基本理解
LLMWiki は、LLMを単なるQ&A相手ではなく、知識ベースの保守者として使う発想。
普通のRAGのように、毎回 raw な資料から答えを再構成するのではなく、
`raw sources -> compiled wiki -> query outputs`
という永続的な中間層を持つ点が特徴。

### 基本の3層
1. raw sources
   - 記事、論文、メモ、画像などの元資料
   - 原則として不変
2. compiled wiki
   - LLMが生成・更新するMarkdown群
   - 要約、概念ページ、比較、索引、派生アウトプットなどを含む
3. schema / instruction file
   - `AGENTS.md` や `CLAUDE.md`
   - 命名規則、更新方針、Ingest / Query / Lint の流れを定義する

### 基本の運用サイクル
- Ingest: 新しいソースを読み wiki を更新する
- Query: wiki を読んで回答する
- File back: 回答自体も wiki に保存して複利化する
- Lint: 矛盾、孤立ページ、欠落概念、古い記述を点検する

### LLMWikiの要点
- 知識を毎回再発見するのではなく、一度「コンパイル」して蓄積する
- 要約、クロスリファレンス、概念整理、矛盾反映をLLMが継続保守する
- 「AIを検索代替として使う」より、「AIに知識インフラを保守させる」発想に近い

## 重要論点
以下の論点を特に重視して整理したい。

1. 人間の思考空間をAI生成物で汚染しないか
   - Obsidianを人間の思考の場として守るなら、AI生成wikiは分離したほうがよい
   - `.agent-wiki/` のような隠しフォルダに置く案が有力

2. Markdown運用で十分か、SQLite等が必要か
   - 中規模なら `index.md` + markdown wiki で十分そう
   - 大規模化すると SQLite + FTS + embedding の構成もありうる
   - ただし個人運用では最初から重くしないほうがよさそう

3. ハルシネーションやドリフトをどう防ぐか
   - persistent wiki は誤りも蓄積しうる
   - source tracing, lint, health check, 必要なら別モデル検証が重要
