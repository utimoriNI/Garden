# 保管庫運用ガイド

このページは、人間がこの保管庫を使うときのための運用案内。

`AGENTS.md` は AI エージェント向けの指示書だが、このページは

- この保管庫で何ができるか
- ノートがどのように流れていくか
- AI にどのように頼めば何が起こるか
- どこまでが AI の作業で、どこからが人間の判断か

を把握するための人間向けドキュメント。

## この保管庫でやっていること

この保管庫では、再利用したい知識断片を蓄積し、

1. 原本ノートを残す
2. `reading-note` に分解する
3. 複数の `reading-note` から Knowledge を作る
4. MOC で見渡せるようにする

という流れで知識管理を行う。

## 主なレイヤー

- `300_Input`
  - Web クリップや各種入力
- `400_Kindle`
  - Kindle ハイライトの原本
- `300_Input/Reading Notes`
  - 再利用しやすい最小単位の `reading-note`
- `600_Knowledge`
  - 抽象化された恒久的な知識ノート
- `110_MOC`
  - 関連ノートを見渡すための MOC
- `.agent-wiki`
  - AI が裏側で使う整理・提案レイヤー

## 重要な考え方

- `🎁Topic/...` は資産カテゴリであり、ノート構造そのものではない
- `reading-note` かどうかはタグではなく frontmatter の `type` で判断する
- 人間のノートと AI 生成物は分離する
- AI は人間ノートを勝手に大きく書き換えない

## 現在運用中のタグ

### `🎁Topic/...`

`🎁Topic/...` は「資産としてどういう種類の内容か」を表す大分類。
ノート構造そのものは表さない。

現在使っている主なタグは次の通り。

- `🎁Topic/Life`
  - 人生の判断、感情、行動原理、対人、自分の指針に関わる内容
- `🎁Topic/Society`
  - 社会構造、仕事、規範、評価制度、文化圧力、能力主義に関わる内容
- `🎁Topic/Learning`
  - 勉強法、上達法、理解の仕方、練習法、認知の鍛え方に関わる内容
- `🎁Topic/PKM`
  - Obsidian運用、ノート整理、知識管理、second brain 的な内容
- `🎁Topic/Tech`
  - 技術、ソフトウェア、生成AI、インターネット理解に関わる内容
- `🎁Topic/Illust`
  - イラスト制作に役立つ内容
- `🎁Topic/DTM`
  - DTM、作詞作曲、音楽制作に役立つ内容
- `🎁Topic/Word`
  - 単語、用語、概念名、法則など「言葉や概念として覚えたいもの」
- `🎁Topic/Rhetoric`
  - 言い回し、比喩、情景、一節、文体など「表現として残したいもの」
- `🎁Topic/Omoro`
  - 面白さ、妙味、ネタ性で残したいもの

補足:

- 1ノートに複数の `🎁Topic/...` が付いてよい
- 高抽象度タグは広めに付け、不要ならあとで外す運用でよい
- 旧 `🎁Topic/Lexicon` は、現在は主に `🎁Topic/Word` と `🎁Topic/Rhetoric` に分けて考える

### `🧩rn/...`

`🧩rn/...` は reading-note 関連の補助タグ。

- `🧩rn/candidate`
  - まだ正式な `reading-note` ではないが、後で `reading-note` 化したい候補

### タグで表さないもの

次はタグではなく frontmatter で管理する。

- `type`
  - 例: `reading-note`
- `source_type`
  - 例: `kindle`, `web`, `legacy`
- `moc`
  - 接続先の MOC
- `status`
  - 運用状態

## 今できること

### 1. Kindle ハイライトから reading-note を作る

`400_Kindle` のノートに `%% pick %%` や `%% group: ... %%` を付けると、
AI が `reading-note` 化を支援できる。

関連スクリプト:

- `scripts/generate_kindle_reading_notes.py`

### 1.5. `300_Input` の引用や発言から reading-note を作る

`300_Input` の Web クリップや動画クリップにも `%% pick %%` や `%% group: ... %%` を付けてよい。
AI はその marker が付いたブロックから `reading-note` を作成できる。

関連スクリプト:

- `scripts/generate_input_reading_notes.py`

### 2. 既存ノートを reading-note 化する

既存ノートに `🧩rn/candidate` を付けると、
AI が `reading-note` schema へ寄せる変換を支援できる。

必要なら本文中に `%% title: ... %%` を書いて、
変換後の reading-note タイトルを指定してよい。

関連スクリプト:

- `scripts/convert_rn_candidate_notes.py`

### 3. reading-note から中抽象度テーマを見つける

現在、`.agent-wiki/theme-discovery/` では
`Life / Society / Learning` を主対象に、`reading-note` から MOC 候補や中抽象度テーマを見つける半自動運用が動いている。

考え方としては、

- `Life`
- `Society`
- `Learning`

は嗜好、価値観、判断、理解の仕方に近いタグなので、LLM wiki の中心的な入力レイヤーとして扱う。

一方で、

- `Word`
- `Rhetoric`
- `Illust`
- `DTM`
- `Tech`
- `Omoro`

などは、現時点では保存目的のスクラップ性が相対的に強いため、LLM wiki の主たるテーマ発見対象にはせず、必要になったときに個別スコープを追加する方針にしている。

この流れでは、

- `reading-note` を束ねる
- 中抽象度テーマを提案する
- MOC 草案を作る
- 人間が良いものだけ `110_MOC` に昇格する

という手順を踏む。

### 4. 既存 MOC に入りそうな reading-note 候補を出す

人間向け MOC を育てるために、
AI は「今ある `reading-note` の中で、この MOC に追加できそうなもの」の候補レポートを出せる。

現在の対象例:

- [[110_MOC/指標に回収される人間の価値_MOC]]

## AI に自然言語で頼めること

この保管庫では、theme discovery 系の処理を自然言語で依頼してよい。

たとえば次のように頼める。

- `reading-note 全体を見て`
- `reading-note 全体を llm wiki に組み込む作業を進めて`
- `全 reading-note の状況を更新して`
- `theme discovery を回して`
- `Life / Society / Learning を llm wiki に組み込んで`
- `Life と Society と Learning を見て`
- `主に考えに関わる reading-note を見て`
- `コアなテーマ発見を更新して`
- `Life 単体で theme discovery を回して`
- `Life x Society の候補を更新して`
- `Life x Learning の候補を更新して`
- `MOC候補レポートを出して`
- `Learning 単体で theme discovery を回して`
- `Society x Learning の候補を更新して`
- `reading-note から今のMOC候補を見直して`

これらを依頼すると、AI は `.agent-wiki/theme-discovery/` の標準ワークフローを実行し、

- スコープ全体のレポート
- 各 MOC 向けの候補レポート
- 最新 run のまとめ

を更新する。

## 現在の theme discovery の対象

現在の基盤スコープは `reading-note` 全体。

対象条件:

- `300_Input/Reading Notes` にある
- `type: reading-note`

その上に、`Life / Society / Learning` のいずれかを持つコアテーマ発見スコープを使っている。

対象条件:

- `300_Input/Reading Notes` にある
- `type: reading-note`
- `🎁Topic/Life` または `🎁Topic/Society` または `🎁Topic/Learning` を持つ

追加で、次の単一タグスコープを扱える。

- `Life only`
- `Society only`
- `Learning only`

対象条件:

- `300_Input/Reading Notes` にある
- `type: reading-note`
- それぞれの対象タグを持つ

さらに、次の交差タグスコープを扱える。

- `Life x Society`
- `Life x Learning`
- `Society x Learning`

対象条件:

- `300_Input/Reading Notes` にある
- `type: reading-note`
- それぞれの対象タグの両方を持つ

## 人間と AI の役割分担

### AI がやること

- `reading-note` を束ねる
- テーマ候補を提案する
- MOC 草案を作る
- 既存 MOC へ入りそうな note 候補をレポート化する
- `.agent-wiki` の index / log / report を更新する

### 人間がやること

- テーマ候補を採用するか判断する
- MOC として正式に残すか判断する
- 既存 MOC との役割分担を決める
- 必要なときだけ `110_MOC` や `600_Knowledge` を更新する

## どこまでが AI 用の作業か

基本的に `.agent-wiki/` までは AI の作業場。

ここでは

- クラスタ
- テーマ候補
- MOC 草案
- 候補レポート

のような中間成果物を作る。

これに対して `110_MOC` や `600_Knowledge` は人間が今後使い続ける正式レイヤー。
AI が作った草案のうち、良いものだけを人間が選んで昇格させる。

## 日常運用の流れ

### 新しい reading-note を作ったとき

1. いつも通り `reading-note` を作る
2. `🎁Topic/...` を付ける
3. 必要なら `theme discovery を回して` と AI に頼む
4. AI が候補レポートを更新する
5. 人間が必要なものだけ MOC に反映する

### 新しく reading-note を作るときのワークフロー

`reading-note` を作る入口は大きく3つある。

1. Kindle ハイライトから作る
2. 既存ノートをそのまま `reading-note` 化する
3. Web クリップや動画クリップの一部分から作る

基本方針は次の通り。

- ノート全体を `reading-note` にしたいなら `🧩rn/candidate` を使う
- ノートの一部分だけを `reading-note` にしたいなら `%% pick %%` や `%% group: ... %%` を使う
- 作成後は `type: reading-note` のノートを `300_Input/Reading Notes` に集める

#### 1. Kindle ハイライトから作る

1. `400_Kindle` の原本ノートを開く
2. 一節ノートにしたい箇所に `%% pick %%` を付ける
3. 複数箇所を1つにまとめたいときは `%% group: ... %%` を付ける
4. タイトルを指定したいときは `%% title: ... %%` を付ける
5. AI に reading-note 作成を依頼する
6. AI が `300_Input/Reading Notes` に生成し、処理済みマーカーを削除する

#### 2. 既存ノートをそのまま `reading-note` 化する

1. そのノート全体が `1ノート = 1断片` になっていることを確認する
2. そのノートに `🧩rn/candidate` を付ける
3. 必要なら `%% title: ... %%` で変換後のタイトルを指定する
4. AI に `reading-note` 化を依頼する
5. AI が最小 frontmatter を追加し、`300_Input/Reading Notes` に集約し、`🧩rn/candidate` を削除する

補足:

- [タグ保存用.md](200_Inbox/タグ保存用.md) の `🧩rn/candidate` はサジェスト用なので変換対象にしない

#### 3. Web クリップや動画クリップの一部分から作る

1. クリップノートは原本として残す
2. 一節ノートにしたい箇所を `1発言 = 1ブロック` や `1引用 = 1ブロック` で書く
3. 抜き出したいブロックに `%% pick %%` を付ける
4. まとめたいブロックには同じ `%% group: ... %%` を付ける
5. タイトルを固定したいときは `%% title: ... %%` を付ける
6. AI に reading-note 作成を依頼する

動画クリップノートでは、次のような情報があると扱いやすい。

- セリフ本文
- タイムスタンプ付きリンク
- 必要なら話者
- 必要なら短いメモ

この場合、ノート全体に `🧩rn/candidate` を付ける必要はない。
一部分だけ抜き出したいので、`%% pick %%` / `%% group: ... %%` を使う。

#### 4. 手動で作る

AI を使わずに自分で作るときは、テンプレートを使って新規作成してよい。

- [[998_Resource/Template/Reading Note]]

手動作成後は、

1. `source_type` を適切に直す
2. `source_container` を必要に応じて入れる
3. `🎁Topic/...` を付ける
4. あとで必要なら `moc` や `topic` を埋める

### MOC 候補を増やしたいとき

1. `Life x Society の候補を更新して` と頼む
2. AI が `.agent-wiki` のクラスタ・テーマ候補・レポートを更新する
3. 良いものだけ正式 MOC 候補として採用する

## 定期メンテ手順

### 週1でやるとよいこと

1. 新しく増えた `reading-note` を確認する
2. 必要なら `theme discovery を回して` と頼む
3. scope report と候補レポートを見る
4. 既存 MOC に入れるべき note を承認する
5. 明らかに有望なテーマ候補があれば、MOC 草案まで進める

### 月1でやるとよいこと

1. 今ある MOC 候補を見直す
2. 既存 MOC と役割が重なっているものを整理する
3. 育ってきた MOC から Knowledge-note 候補を切り出す
4. `.agent-wiki` 側のテーマ候補で、不要になったものや弱いものを棚卸しする
5. 必要なら theme discovery の対象やルールを見直す

### MOC が増えてきたとき

1. `moc_registry.json` に新しい MOC 用 config を追加する
2. 標準サイクルにその MOC の候補レポートを組み込む
3. theme discovery を回したときに scope report と各 MOC レポートが同時に更新される状態にする

### theme discovery の対象を広げたいとき

今は `Life x Society` を標準対象にしているが、必要なら将来的に

- `Learning x Word`
- `Society x Word`
- `Life x Rhetoric`
- `Society only`

のような別スコープを追加できる。

その場合は、

1. 新しい scope config を作る
2. 必要なら MOC 候補を追加する
3. registry や workflow を更新する

という順で広げる。

## 注意点

- `.agent-wiki` の内容は自動で正式ノートになるわけではない
- AI の提案は候補であり、最終判断は人間が行う
- `110_MOC` や `600_Knowledge` への反映は、意図を持って行う
- タグは資産カテゴリであり、構造タイプとは分けて考える
