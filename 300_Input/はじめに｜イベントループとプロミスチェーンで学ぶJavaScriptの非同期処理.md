# 第 1 章 API を提供する環境と実行メカニズム

## 非同期 API と環境

### 非同期処理の解説で見過ごされがちな話

- 非同期処理は Promise や async/await といった ECMAScript の機能が注目されがち
  - 非同期処理を解説するのに`setTimeout()`という API を使ってしまう
- 非同期処理は「非同期 API+コールバック関数」や「await 式+非同期 API」という形が多い
  - 非同期処理は**ECMAScript と非同期 API を組み合わせて使われる**

### API の機能を提供する環境

- JavaScript には実行するための環境があり、各環境には違いがある
  - 相違点として顕著な物が、環境が提供する機能である API
- ブラウザ環境が提供する API は MDN において**Web API**と呼ばれている
  - 実態としては**Browser API**という表現が近く、ブラウザ環境によって提供されている
- Web 上で提供される何らかのサービスの API は MDN において**サードパーティ API**と呼ばれている
- Node、Deno などで提供されている API は便宜的に**Runtime API**と呼ぶ
  - `setTimeout()`などが他の API と同じように存在しているが微妙に実装が異なる

### 非同期処理の目的と仕組み

- 非同期処理は、**なぜ実行のタイミングをずらす必要があるのか？**

#### 非同期 API の`fetch()`メソッドで考える

- `fetch()`は引数に URL を指定することでネットワーク接続をしてデータ取得ができる
  - ネットワーク接続は応答までに時間がかかる
- JavaScript はシングルスレッド言語であり、**単一のコールスタックで実行される**
  - つまり、メインスレッドが長時間占有されるとその間は何もできなくなる →**ブロッキング**
  - ブラウザならクリックやテキスト選択などもできなくなる
  - これを避けるために実行のタイミングをずらす必要がある
- そのため、時間がかかる API 処理には JavaScript エンジンは関与せずに、そのエンジンがある環境が処理を行う
  - データ取得が完了したら、コールスタックにコールバック関数の形で**取得データと共に通知させて**処理を行う

#### JavaScript は並列処理をしていないのか？

- JavaScript はシングルスレッド言語だが、**ブラウザの環境において複数の処理を並行処理できる**

```js
// コールバック関数をアロー関数で定義
setTimeout(() => {
  console.log("[A] 並列的なタイマー");
}, 1000);
console.log(
  "上のタイマー処理が終わっていないのにコンソールへ出力し、次のタイマーを起動"
);

setTimeout(() => {
  console.log("[B] 並列的なタイマー");
}, 1500);
console.log(
  "上のタイマー処理が終わっていないのにコンソールへ出力し、次のタイマーを起動"
);

setTimeout(() => {
  console.log("[C] 並列的なタイマー");
}, 500);
console.log("上のタイマー処理が終わっていないのにコンソールへ出力");

console.log("タイマーを起動しているのにコンソールに出力ができてしまう");
```

```
❯ deno run parallelTimer.js
上のタイマー処理が終わっていないのにコンソールへ出力し、次のタイマーを起動
上のタイマー処理が終わっていないのにコンソールへ出力し、次のタイマーを起動
上のタイマー処理が終わっていないのにコンソールへ出力
タイマーを起動しているのにコンソールに出力ができてしまう
[C] 並列的なタイマー
[A] 並列的なタイマー
[B] 並列的なタイマー
```

- JavaScript はシングルスレッドだが、環境が JavaScript エンジンだけでなく様々な機能を持っている
  - Chrome は以下のような複数のプロセスを持ってる
    - Browser process
    - Renderer process
    - GPU process
    - Plugin process
    - Extensions process
    - Utility process
  - JavaScript のシングルスレッドは、Chrome においては Render proces の Main Thread であり、`fetch(url)`の根とワーク接続やリクエストの処理を行っているのは Browser process の Network thread

#### 非同期処理の目的と仕組み

- 目的：環境がバックグラウンドで並列的に作業している間も**メインスレッドをブロッキングすることなく別の作業を続けられるようにすること**
  - 配置とタイミングがずれて処理されるのが非同期処理だが、これは結果であって目的ではない
- 仕組み：環境が提供する API を介して時間がかかる処理を環境に委任し、完了したら JavaScript のメインスレッドに処理結果と共に通知することで関連する作業をコールバックなどの形式で行う

##### 非同期処理は複数のカテゴリに分解できる

1. `setTimeout()`や`fetch()`などの非同期 API：起動後は環境によって並列的にバックグラウンドで処理される
2. 非同期のシンタックスで書かれた関数（ECMAScript の関数）：非同期 API の処理結果を使った後に何か処理する作業で、Callback や Promise chain、async/await で記述される

- 1 の非同期 API は非同期処理というより**並列処理**とみなすほうが理解しやすい
