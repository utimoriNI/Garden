# 第1章 APIを提供する環境と実行メカニズム

## 非同期APIと環境
### 非同期処理の解説で見過ごされがちな話
- 非同期処理はPromiseやasync/awaitといったECMAScriptの機能が注目されがち
	- 非同期処理を解説するのに`setTimeout()`というAPIを使ってしまう
- 非同期処理は「非同期API+コールバック関数」や「await式+非同期API」という形が多い
	- 非同期処理は**ECMAScriptと非同期APIを組み合わせて使われる**

### APIの機能を提供する環境
- JavaScriptには実行するための環境があり、各環境には違いがある
	- 相違点として顕著な物が、環境が提供する機能であるAPI
- ブラウザ環境が提供するAPIはMDNにおいて**Web API**と呼ばれている
	- 実態としては**Browser API**という表現が近く、ブラウザ環境によって提供されている
- Web上で提供される何らかのサービスのAPIはMDNにおいて**サードパーティAPI**と呼ばれている
- Node、Denoなどで提供されているAPIは便宜的に**Runtime API**と呼ぶ
	- `setTimeout()`などが他のAPIと同じように存在しているが微妙に実装が異なる

### 非同期処理の目的と仕組み
- 非同期処理は、**なぜ実行のタイミングをずらす必要があるのか？**

#### 非同期APIの`fetch()`メソッドで考える
- `fetch()`は引数にURLを指定することでネットワーク接続をしてデータ取得ができる
	- ネットワーク接続は応答までに時間がかかる
- JavaScriptはシングルスレッド言語であり、**単一のコールスタックで実行される**
	- つまり、メインスレッドが長時間占有されるとその間は何もできなくなる→**ブロッキング**
	- ブラウザならクリックやテキスト選択などもできなくなる
	- これを避けるために実行のタイミングをずらす必要がある
- そのため、時間がかかるAPI処理にはJavaScriptエンジンは関与せずに、そのエンジンがある環境が処理を行う
	- データ取得が完了したら、コールスタックにコールバック関数の形で**取得データと共に通知させて**処理を行う

#### JavaScriptは並列処理をしていないのか？
- JavaScriptはシングルスレッド言語だが、**ブラウザの環境において複数の処理を並行処理できる**
```js
// コールバック関数をアロー関数で定義
setTimeout(() => {
  console.log("[A] 並列的なタイマー");
}, 1000);
console.log("上のタイマー処理が終わっていないのにコンソールへ出力し、次のタイマーを起動");

setTimeout(() => {
  console.log("[B] 並列的なタイマー");
}, 1500);
console.log("上のタイマー処理が終わっていないのにコンソールへ出力し、次のタイマーを起動");

setTimeout(() => {
  console.log("[C] 並列的なタイマー");
}, 500);
console.log("上のタイマー処理が終わっていないのにコンソールへ出力");

console.log("タイマーを起動しているのにコンソールに出力ができてしまう");
```
```
❯ deno run parallelTimer.js
上のタイマー処理が終わっていないのにコンソールへ出力し、次のタイマーを起動
上のタイマー処理が終わっていないのにコンソールへ出力し、次のタイマーを起動
上のタイマー処理が終わっていないのにコンソールへ出力
タイマーを起動しているのにコンソールに出力ができてしまう
[C] 並列的なタイマー
[A] 並列的なタイマー
[B] 並列的なタイマー
```
- JavaScriptはシングルスレッドだが、環境がJavaScriptエンジンだけでなく様々な機能を持っている
	- Chromeは以下のような複数のプロセスを持ってる
		- Browser process
		- Renderer process
		- GPU process
		- Plugin process
		- Extensions process
		- Utility process
	- JavaScriptのシングルスレッドは、ChromeにおいてはRender procesのMain Threadであり、`fetch(url)`の根とワーク接続やリクエストの処理を行っているのはBrowser  processのNetwork thread

#### 非同期処理の目的と仕組み
- 目的：環境がバックグラウンドで並列的に作業している間も**メインスレッドをブロッキングすることなく別の作業を続けられるようにすること**
	- 配置とタイミングがずれて処理されるのが非同期処理だが、これは結果であって目的ではない
- 仕組み：環境が提供するAPIを介して時間がかかる処理を環境に委任し、完了したらJavaScriptのメインスレッドに処理結果と共に通知することで関連する作業をコールバックなどの形式で行う

##### 非同期処理は複数のカテゴリに分解できる
1. `setTimeout()`や`fetch()`などの非同期API：起動後は環境によって並列的にバックグラウンドで処理される
2. 非同期のシンタックスで書かれた関数（ECMAScriptの関数）：非同期APIの処理結果を使った後に何か処理する作業で、CallbackやPromise chain、async/awaitで記述される
- 1の非同期APIは非同期処理というより**並列処理**とみなすほうが理解しやすい
