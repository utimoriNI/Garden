# リーダブルコード
## 1章　理解しやすいコード
- コードを理解するまでにかかる時間を最小化することを目指す
	- 理解するまでにかかる時間を短くすることは、コードを短くすることよりも重要
- 自分しか見ないコードだったとしても、数ヶ月後の自分がそのコードを見る時はコードの意味を覚えていない
## 2章　名前に情報を詰め込む
### 明確な単語を選ぶ
- 例えば`get`はあまり明確ではない
	- `def GetPage(url)`はどこからページを取ってくる？
- 仮にインターネットから取ってくるのであれば、`FetchPage()`や`DownloadPage()`の方が明確
- ツリーにおける`Size()`は何を意味している？
	- ツリーの高さ、ノードの数、ツリーのメモリ消費量などが考えられる
	- それぞれ、`Height()`、`NumNodes()`、`MemoryBytes()`が適当だろう

### tmpやretvalなどの汎用的な名前を避ける
- tmp・retval・fooなどの名前は、名前のことを考えていないことを表す名前
- tmpという名前は、生存期間が短く、一時的な保管だけが役割の変数に使う
	- tmpを使うときも、`tmp_file`などと一時保存する内容を示すと良い

##### ループイテレータ
- `i・j・k・iter`などの名前は汎用的名前だが、これを使うだけでイテレータという意味を持たせることができるので使っても問題ない
	- ただ、もっといい名前をつけられる場合がある
	- 複数のイテレータを同時に使うとき（forの入れ子など）は、イテレータの名前を明確にした方が良い
```javascript
for(int i=0; i<clubs.size(); i++)
	for(int j=0; j<member.size(); j++)
		if(clubs[j] == member[i]) // インデックスが逆

for(int club_i=0; club_i<clubs.size(); club_i++)
	for(int member_j=0; member_jj<member.size(); member_jj++)
		if(clubs[clubs_i] == member[member_j]) // インデックス分かりやすい
```

### 名前の長さを決める
##### スコープが小さければ短い名前でいい
- 変数や関数が使われる場所が少なければ、多くの情報を名前に詰め込む必要はない
	- 型や初期値などが一覧できるから

### 名前のフォーマットで情報を伝える
- GoogleのC++におけるフォーマット
	- クラス名は`CamelCase`
	- 変数は`lower_separated`(小文字をアンダースコアで区切る)
	- 定数は`kConstantName`
	- クラスのメンバ変数は`offset_`のようにアンダースコアを最後に置く
		- ローカル変数と区別するため
- JavaScript: The Good Partsにおけるフォーマット
	- コンストラクタは大文字で初めて普通の関数は小文字で始める
	- jQueryのライブラリ関数を呼び出した時には変数名の頭に$をつける
```javascript
var x = newDatePicker(); //コンストラクタ関数
var y = pageHeight() //普通の関数
var $all_images = $(“img”)
```
- HTML・CSSのフォーマット
	- idの区切り文字にはアンダースコア、classの区切り文字にはハイフンを使う

## 誤解されない名前
**鍵となる考え：名前が「他の意味と間違えられることはないか？」と何度も自問自答する**

#### 例：filter()
- データベースの問い合わせ結果を処理するとき
	- `results = Datbase.all_objets.filter(”year <= 2011”)`
	- この`results`には「year <= 2011」のオブジェクトと、「year <= 2011」でないオブジェクトのどちらが含まれているか？
- 選択するのであれば`select()`が良いし、除外するのであれば`exclude()`にした方が良い

#### 例：Clip(text, length)
- 段落の内容を切り抜く関数
```
# textの最後を切り落として「…」をつける
def Clip(text, length):
```
- Clipの動作は2つ考えられる
	- 1. 最後から`length`文字削除する（remove）
	- 2. 最大`length`文字まで切り詰める(truncate)
- `length`という名前も曖昧で、バイト数・文字数・単語数といった解釈ができる
	- `max_char`が良い

#### 範囲の指定（first / last, begin / end）
- first / lastは最後を含む時に使う
- begin / endは最後を含まない時に使う


## 5章 コメントすべきことを知る
**鍵となる考え：コメントの目的は、書き手の意図を読み手に知らせることである**

### コメントするべきでは「ない」こと
- 新しい情報を提供しない、または読み手がコードを理解しやすくならないコメントはいらない
	- `class Account{} //Accountクラスの定義`
	- 上記のコメントは不要
- コメントのためのコメントをしない
	- 関数に対してコメントをつけるときは、関数宣言とほぼ同じコメントを書いても意味がない
	- 関数の処理がわかるようなコメントを書くべき

### 自分の考えを記録する
- なぜそのような処理にしたのかを書く
	- `// このデータだとハッシュテーブルよりもバイナリツリーの方が早かった`
	- `左右の比較よりもハッシュの計算コストが高い`
- コードの欠陥にコメントをつける
	- `// TODO: もっと高速なアルゴリズムを使う`
	- `// TODO: JPEG以外のフォーマットに対応する`

### ハマりそうな罠を告知する
- 関数やクラスを文書化する時には、「このコードを見てびっくりすることはなんだろう？どんな風に間違えて使う可能性があるだろう？」と問いかける
	- 実装の詳細を告知する：`// メールを送信するサービスを呼び出す（1分でタイムアウト）`
	- 誤った使い方を防ぐ：`//実行時間はO(タグの数 * タグの深さの平均)なので、ネストの深さに気をつける`

### ライターズロックを乗り越える
- ライターズロック=行き詰まってしまい、文章が書けないこと
- 自分の考えていることをとりあえずコメントに書き出してみよう
	- ある関数を作っていて、「ヤバい。これはリストに重複があったら面倒なことになる」と思ったら、それをそのままコメントに書いてみる
	- そのコメントが残す必要のあるものならば、丁寧な言い方に書き換える
		- 上記の例なら、「このコードはリストの重複を処理できません（実装が難しいので）。」
