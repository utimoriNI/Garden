## はじめに

最近、スマホのChatGPTアプリからCodex経由で **パソコンを操作できる** ようになりました。

これを使って始めたのが、 **「技術ノート」** の作成。 **メモる速度で記事化する** という作業。  
疑問が浮かんだその瞬間、Codexに指示して記事化を始めるようになって、学習体験がすごく良くなりました。  
移動中に音声入力で「質問〜」と投げることも多いです。

![スマホに話しかけた疑問がCodexで記事化され、GitHubに蓄積され、あとから見つかる流れ](https://static.zenn.studio/user-upload/deployed-images/38119faca31693111f1f3f2a.png?sha=fbb7cdfdfb44681a5c78db0898b5f749db87f0de)

- 人との会話で知った情報が気になったとき
- 電車で見た記事の深掘りをしたいとき
- 寝る前にふと気になったことを思い出したとき

今日から誰でも始められて、効果もすぐ体感できます。この記事では、私が実際に回している **スマホ発・即記事化** 運用を紹介します。成果物の例として、 [技術質問ノート](https://ttaniguchi.github.io/ai-teacher/) を GitHub Pages で公開しています。

## この体験が新しい！

「調べる」と「残す」がかなりシームレスになり、パソコンの前に縛られなくなりました。  
また、オープンな記事のブックマークとは違い、自分なりのメモを積み足したり、複数の記事をサマリーできるのもメリットです。

私の経験上、スマホで調べものをすると、その場では分かった気になっても深く記憶に残らないことがよくありました。  
忘れてから改めて調べると同じ記事に当たらず、読み直しになって効率の悪い学習になりがちでした。

その場で記事化することで調べ直しは激減し、忘れたときも同じところに情報を求めに行く癖がつきました。「その場の知識」が「資産」になっていくのを実感できます。

## 「技術ノート」構築方法

## 自宅でCodexを起動したパソコンを常時待機

スマホで疑問を投げると、自宅のパソコンでCodexがHTMLページを追加してくれます。コードの確認はスマホのCodex上、もしくは帰宅後にパソコンで行います。パソコンがスリープしないよう、設定をしておきましょう。

![](https://static.zenn.studio/user-upload/deployed-images/3e31a57c8e099d82461a4627.png?sha=4604ab692f7c24de2a213583cef6fb963e8d0e2e)

## 静的HTMLサイトのプロジェクトを1つ用意

構成はいたってシンプルです。

```
notes/
├── .github/workflows
│   └── static.yml # GitHub Pages へのデプロイ処理
├── styles/
│   └── site.css
├── images/
│   └── [画像・SVG等]
├── [記事・カテゴリ1]/
├── [記事・カテゴリ2]/
├── [記事・カテゴリ3]/
├── .gitignore # 静的HTMLページに必要なファイルのみ許容
├── AGENTS.md # 秘匿情報の記載禁止など、記事のルールを記載
├── README.md
└── index.html # ドキュメントルート
```

ビルドは不要、HTML/CSS/SVGと画像だけ。GitHub Pages にそのまま載せられる形です。  
1ページ1テーマで `index.html` に列挙して、カテゴリを頼りに辿れるようにしています（ [公開中の例](https://ttaniguchi.github.io/ai-teacher/) ）。  
`README.md` にはリポジトリの概要だけ載せ、運用ルールの詳細は `AGENTS.md` に寄せています。

## 記事化スキルを用意

スキルは Codex のユーザーディレクトリに保持しているので Git 管理の対象外ですが、シンプルなルールで不便はありません。  
「私が『質問』で始める文章で指示した際は、HTMLで図解などを含むわかりやすい記事を作って、コミットまでしてね」的な内容を相談しながら書いてもらいました。

## 出先で読めるホスティング

Codex経由でコードをプレビューすることはできますが、せっかくHTMLで構築しているのでブラウザで読みたいです。私は **GitHub Pages** へのデプロイが手っ取り早く感じたので、そちらを利用しています。

私が利用しているプランだと、GitHub Pages を利用するには公開リポジトリにする必要がありました。リポジトリの内容はネットで調べたことのまとめなので秘匿性はなく、私は公開リポジトリで運用しています。ただ、AIによる生成物がいきなり公開されるのは危険なので、以下のガードレールを敷いています。

- GitHub へのプッシュはコード差分の目視後に都度承認を行っている（人間がガードする）
- 秘匿情報の扱いは前述の `AGENTS.md` と `.gitignore` で担保する（JSON や YAML などは Git 管理外）

上記は私の運用例ですが、HTMLとCSSだけで動く極めてシンプルな静的ファイル構成なので、ホスティング先はどこでも大丈夫です。

- Google Driveなどのクラウドストレージに放り込んで自分だけで見る
- 任意のプライベートなホスティング環境にデプロイする

とはいえOSSの考え方であれば、公開することで記事の間違いに修正PRをいただける可能性はあるかもしれませんね（笑）

## AI任せで内容は大丈夫？

この運用では本文をすべてAIが生成します。なので一次情報は調べたほうがよいでしょう。ただし、一次情報の重要性自体は、他人のブログを読むときにも同じだと思っています。

記事生成のスキルには一次情報が必須であると記載して、HTMLページの末尾に参照URLが必ず載せられるような形で運用するのが良いと思います。

## まとめ

今回の手法は、完璧なものを構築するのではなく、情報のピックアップを加速するために AI を活用した事例です。そのうえでスマホと AI は非常に相性がよく、特に音声を介したときは、パソコンの前でキーボードを叩くよりはるかに効率的です。

ぜひ、「スマホで見る」から「スマホで資産化する」へ体験を向上させてみましょう。この差は、想像以上に大きいです。
